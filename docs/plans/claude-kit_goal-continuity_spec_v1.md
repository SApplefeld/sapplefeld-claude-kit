# Goal Continuity: /kit-goal Arming and the Leash Across Session Swaps

Status: In Progress
Commit Model: Commit-and-Push

## Goal

Native `/goal` enforcement and relay compaction now compose: the goal template's handoff carve-out (executing-work, The completion contract) lets the evaluator approve a boundary stop, so a relay-armed compaction ends the turn cleanly instead of racing the Stop hook. That wording fix shipped with this spec's changeset (executing-work goal template and step 8 branches, compact-session relay precondition).

Two gaps remain, and this effort closes them:

1. **The leash is discontinuous across the swap.** Goal state is session-scoped, so the compacted successor starts goalless. Until it is re-armed, the successor runs on the completion contract alone, exactly the "stopped after one big chapter" exposure `/goal` exists to prevent.
2. **Arming is manual boilerplate.** The canonical goal text must be composed and pasted per run. The target ergonomic: `/kit-goal docs/plans/<plan>.md` arms the run in one line.

**Live-fire follow-up (added 2026-07-17).** An unsupervised live fire, a `/kit-goal`-armed run on ASR's EleosMcp repo crossing a 674k boundary, surfaced two more gaps that Sections 6 and 7 close: the leash blocks a turn that legitimately awaits a background-dispatched implementer (Section 6), and the resume relay cannot resolve the successor's window on a remote/ConPTY host, so the compaction never sheds context and the run grows unbounded (Section 7). Chapter 11 records the incident and the diagnosis.

## Facts (code.claude.com/docs/en/goal.md)

- `/goal` is a wrapper around a session-scoped prompt-based Stop hook. At each stop attempt, the condition plus the conversation go to the configured small model (Haiku default), which returns yes or no.
- Conditional clauses in the condition text are honored (bounding clauses such as "or stop after 20 turns" are documented).
- `/goal clear` clears the goal (aliases: stop, off, reset, none, cancel). Only the evaluator declares completion; the session cannot end its own goal.
- A goal carries over when resuming the same session. A compacted successor is a new session ID, so inheritance is not expected there; Section 1 confirms.
- A custom slash command cannot alias a built-in directly: command files expand to prompt text, they do not execute other commands. The two viable `/kit-goal` routes are the SlashCommand tool (if it can invoke `/goal`) and a kit-native mechanism that does not need `/goal` at all.

## The Section 3 fork (decided 2026-07-16: A, kit-native)

Scott chose fork A: a deterministic kit-owned implementation of the plan methodology, keeping native `/goal` in reserve for goals that are not plan-based. Fork B stands only as the fallback if A's Stop-hook UX fails live fire.

**A. Kit-native goal (recommended).** `/kit-goal <plan>` writes a project-scoped goal state file; a kit Stop hook enforces it deterministically: allow the stop when (a) the armed plan's Status is Complete or the plan is archived, (b) the last assistant message leads with `BLOCKED:`, or (c) a resume-relay request was written in the last few minutes; otherwise block with a reason naming the plan. Why it wins: the successor session inherits the leash structurally (same project, same hook), no typed keystrokes, arming is fully automatic, deterministic conditions cannot be sweet-talked, and it works wherever hooks run, Desktop included. Costs and open design points: it duplicates a native feature and the kit maintains it; the blast is project-wide, so the hook must scope to sessions actually working the armed plan (candidate predicate: the session transcript references the plan path); it must handle `stop_hook_active` without looping; and the clause-(c) window must tolerate the watcher archiving `request.txt` within its 10-second poll (accept a recent `processed\` entry, or a dedicated boundary marker).

**B. Watcher goal-line extension.** The relay request contract gains an optional goal line; after typing the resume, the watcher types `/goal <condition>` into the successor, then the continue prompt. Why it loses: it adds two typing stages to a keystroke path whose focus losses are already hard failures, it covers only the relay path (a crash or native auto-compaction still drops the leash), and arming stays manual. It remains the fallback if A's Stop-hook UX proves troublesome in live fire.

With A, the goal template's single owner moves to the `/kit-goal` command and executing-work's template block becomes a pointer to it, per the one-owner rule.

## Locked contract (as-built, decided 2026-07-16)

Fork A's mechanism is deterministic (no LLM evaluator). Sections 2, 3, and 5 all build to this shared contract:

- **Goal state file:** `<repo>/.kit/goal-state.json` (gitignored, project-scoped so the successor session inherits it via shared cwd). Schema: `{ "plan": "<repo-relative forward-slash plan path>", "condition": "<canonical condition text>", "armedAt": "<ISO-8601>", "boundSession": "<session uuid>" | null }`. Written atomically (tmp + rename); arm and re-arm both write `boundSession: null` (re-arm is the rebind path when a bound session died without a handoff).
- **Shared library:** `plugins/claude-kit/hooks/kit-goal-lib.js` (Node core only, CommonJS, never throws), the single owner of the canonical condition string (`composeCondition`), plus `readGoal` / `armGoal` / `bindSession` / `clearGoal` / `planHead`. The lib, not the skill prose, is the single source of the condition text.
- **CLI entry:** `plugins/claude-kit/hooks/kit-goal.js` (`arm <plan>` | `clear` | `status`), invoked by the skill.
- **`/kit-goal` skill:** `plugins/claude-kit/skills/kit-goal/SKILL.md`, arming and clearing UX; executing-work's template block becomes a one-line pointer to it (the one-owner move).
- **Stop hook:** `plugins/claude-kit/hooks/kit-goal-stop.js`, wired in `hooks.json`. Fires on every Stop but is a strict no-op unless a goal is armed. Allow order: (0) no state file → allow; (0b) scoping is a session-identity binding: while `boundSession` is null, the first stopping session whose slash-command arguments (`<command-args>` spans in genuine user entries, i.e. the `/kit-goal` arming invocation; never plain prose mentions, assistant text, attachments, tool_results, sidechain turns, harness-injected `isMeta` entries, or local-command stdout) name the armed plan path claims the binding (last-writer-wins between simultaneous claimers, converging at the next stop) and is enforced; once bound, only the bound session is enforced and any other session is allowed regardless of what its transcript mentions, except a stopping session reachable from `boundSession` through the compaction ledger's sourceSessionId-to-destinationSessionId chain (multi-hop, cycle-guarded, case-insensitive compares), which rebinds to itself as the chain's legitimate successor (the manual-resume inheritance path); a failed bind or rebind write still enforces that stop and retries naturally at the next one; (a) plan Status is Complete or the plan file is gone (archived) → auto-clear and allow; (b) last assistant message leads with `BLOCKED:` → allow, with reads-tolerate-the-writer guards because the harness can still be appending the turn's final entries when the Stop hook runs: a read that does not resolve the last turn (no lead found, or a partial mid-append/cap-cut final line) is re-read briefly (150ms/350ms; env-overridable for tests, clamped), a persistent no blocks, and a persistent partial tail is indeterminate → allow; (c) a resume-relay handoff for this plan is recent (live `request.txt`, or newest `processed\*.txt`, mtime within the window AND its content names the armed plan's repo-relative path, so a same-basename plan in another repo never matches) AND its destination session UUID (request line 1) is not the stopping session → allow, rebinding `boundSession` to that destination so the successor holds the leash from its first turn (a failed rebind write still allows; the ledger genealogy lets the successor claim later); the newest handoff naming the plan decides, so one whose destination IS the stopping session ends the check without allowing rather than falling through to an older handoff, while handoffs naming other plans are skipped; else block with a reason naming the plan. Any error → allow (never trap the session). The conditions re-evaluate on every stop attempt, including inside a stop-hook continuation (`stop_hook_active` is not an early-exit): loop safety is the harness's documented consecutive-block cap (eight blocks without progress, `CLAUDE_CODE_STOP_HOOK_BLOCK_CAP`), which releases a genuinely stuck session with a visible warning.
- **Clause-(c) window:** 5 minutes, tolerating the watcher's 10-second poll archiving `request.txt` to `processed\`. Tying the match to the plan path (not just recency) tightens it against a concurrent unrelated relay on the same machine, and the destination-UUID exclusion closes the successor grace hole: the request that resumed a session never doubles as that session's own license to stop inside the window. This leaves the compact-session relay write path untouched (no added boundary-marker step).
- **Session-start surfacing:** `session-start.js` emits "kit goal armed for &lt;plan&gt;" when a goal is armed, so no session is surprised by the hook.

Section-boundary reallocation from the brainstorm sketch: the `/kit-goal clear` UX lands in Section 2 (it owns the whole skill + CLI surface); Section 3 owns the hook's auto-clear-on-Complete. Clean disjoint file ownership: Section 2 = lib + CLI + skill + executing-work pointer; Section 3 = Stop hook + hooks.json + session-start.js.

### 6. Leash / async-dispatch reconciliation
Model: inline (behavior-shaping skill prose plus a transcript-only investigation; design-entangled)
The armed leash blocks a turn that ends to await a background-dispatched implementer. The async-dispatch orchestration model, a `run_in_background` agent that ends the turn for a completion notification, collides with the leash, which reads that turn-end as a stop. The completion contract already owns the governing principle for build and test gates ("a wait is not a stop, poll it in-turn"); it was never extended to agent dispatch. Two parts:

- **6a (guidance, this changeset).** Extend executing-work's completion contract so awaiting a dispatched subagent falls under the same rule: under a leash, keep the turn alive. Dispatch the critical-path implementer synchronously (`run_in_background: false`) so its whole run is one in-turn call; for a genuine parallel fan-out, poll the agents' output files in-turn as with a build gate. Never end the turn on a completion notification while a leash is armed. Add a pointer from step 1's dispatch mechanics. Run the wording through writing-skills (behavior-shaping edit).
- **6b (hook probe, transcript-only).** Characterize the pending-work record shapes from ASR's transcript (`Async agent launched successfully`, `Command running in background with ID`, the `<task-notification>` completion) and design a hook-side "pending background work → allow" detection against them, then record a go/no-go on building it. The one live unknown, what the Stop-hook stdin payload exposes about pending tasks, is flagged rather than assumed; the transcript-tail signal is the fallback the hook would key on. 6a is the floor; the hook change is built only if the probe shows it both warranted and feasible.

Acceptance: executing-work carries the wait-is-not-a-stop-for-dispatch rule with the foreground-dispatch mechanism named; 6b yields a documented go/no-go with the record shapes and the flagged live gap. Do NOT patch clause (c) of the Stop hook: the live block it produced (below) was correct.

### 7. Relay dynamic window targeting
Model: opus (capture rewrite plus the compact-session step plus tests), with a supervised ASR verify gate before the build
The resume relay's per-request window target (request line 4, `ahk_id <hwnd>`) is produced by `capture-window.ps1`, which resolves the host window via `GetConsoleWindow` → `GetAncestor`. On a remote or ConPTY host (ASR runs through Remote Desktop) `GetConsoleWindow` returns zero, so line 4 is omitted and every relay falls back to the static `window.txt`, which is read once at watcher startup and hand-maintained. A stale fallback holds a prior session's title (ASR's held "MCP: Demo Build"), so the watcher types into the wrong window or the >1-match guard refuses, the handoff archives to `failed/`, and the run continues uncompacted and grows unbounded. Fix:

Design B (chosen 2026-07-17): inject a per-relay-unique marker into the window title, then resolve the window by that marker. Design A (read the clean session name before the engine relabels, exact-match) is the reserved fallback if B's title-update proves unreliable.

- **Inject a unique marker, then resolve it.** After the engine returns the destination id, compact-session relay mode appends a `custom-title` to the SOURCE transcript embedding a per-relay-unique marker derived from the destination UUID (guaranteed unique, unlike a shared session name or the plain `[UNCOMPACTED]` tag), keeping the resume-picker's `[UNCOMPACTED]` affordance in the same title (e.g. `[UNCOMPACTED] <name> {relay:<dest8>}`). This reuses the exact `appendCustomTitle` mechanism the engine already drives for `[UNCOMPACTED]` (`compact-cli.ts:121-128`), which is confirmed to reach the live window title (the continued `ad25b9fe` session shows its relabel; only the update latency is unmeasured, hence the gate below).
- **Rewrite `capture-window.ps1`** to take the unique marker and poll `EnumWindows` (console-independent, since the ASR probe showed `GetConsoleWindow` is unreliable in the nested compaction-time call) for a bounded window (~10s) for the one Windows Terminal window whose title contains the marker, emitting that window's `ahk_id`. A timeout or a >1 match exits 1 and falls back to `GetConsoleWindow` → `GetAncestor` and then `window.txt`, so B degrades to today's behavior when the title has not updated. The watcher and the request contract are unchanged: capture still emits `ahk_id <hwnd>`, so no free-form title is trusted at the keyboard.
- **Lifecycle.** On a successful resume the successor loads and Claude Code retitles the window, clearing the marker; on a failed resume the marker persists in the title (as `[UNCOMPACTED]` does today), an accepted and informative pollution. The marker is derived from the destination UUID already on request line 1, so it leaks no new information.
- `window.txt` demotes to a last-ditch fallback for an unnamed or tabbed session; the common path needs no manual maintenance.

Constraint (evidenced by the ASR probe, 2026-07-17): the target session must be NAMED and occupy its own Windows Terminal window, never a tab. Tabs are invisible to `EnumWindows`: a multi-tab WT window enumerates as a single HWND carrying only the active tab's title, so a session in a background tab can be neither resolved nor typed into (the watcher activates a window and types into whatever tab is forward, and `GetConsoleWindow→GetAncestor` resolves to the same window, so no method reaches a background tab; this is a hard property of the relay, not of this fix). An unnamed session shows a generic `★ Claude Code` window title with no unique anchor. Both cases (unnamed, or tabbed) fall back to the console path and then `window.txt`, exactly as today; title resolution is the reliable path only for a named, own-window session, which is the autonomous-run setup. This aligns with the watcher's existing exactly-one-match guard.
ASR pre-build gate for B (the latency test): confirm on a throwaway ASR session that an external `appendCustomTitle` updates the live WT window title within capture's poll window (~10s). The earlier `asr-window-probe.ps1` runs already established the surrounding facts: `EnumWindows` resolves a WT window by title from inside an ASR tool process, an unnamed session's window is a generic `★ Claude Code` (fallback territory), and a tabbed window exposes only its active tab (the one-session-per-window constraint above). If the latency test fails (the append does not reach the live title in bounded time), B is infeasible for live capture and Section 7 falls back to Design A.
Acceptance: on ASR, a boundary compaction writes a request whose line 4 is a title-resolved `ahk_id`, the watcher types into the correct window, and the successor resumes with no `window.txt` edit.

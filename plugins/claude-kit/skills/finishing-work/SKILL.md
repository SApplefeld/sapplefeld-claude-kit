# Finishing Work

An effort is not done when the last section compiles. It is done when behavior is verified, security is reviewed, documentation matches reality, and the plan doc is closed. Run these steps in order; steps 2 to 4 may be dispatched in parallel after step 1 passes.

When per-section reviews already cleared parts of the changeset, tell the finishing reviewers what those passes covered and what has changed since, so they spend their budget on cross-section cohesion and on deltas rather than re-deep-reviewing unchanged, already-cleared code. Eliminate true duplication, not coverage. For a small effort that had no meaningful per-section reviews (a few files, one short pass), a single combined adversarial and security pass is enough; do not manufacture separate passes for a handful of files.

In a session running below fable, dispatch the security review (step 2) and the final adversarial review (step 3) with the explicit `fable` model override by default: the fresh-eyes strong-model verdict over the whole changeset is the oversight that makes plan-covered implementation safe. A recorded `Fable Spend: none (cost hold)` in the spec header keeps them at the session model.

## Steps

1. **QA verification.** Dispatch the `qa-verifier` agent with the spec path: full build, full test suite, and every acceptance criterion checked with evidence. Any FAIL: fix and re-run before proceeding. Do not rationalize a failing criterion as "close enough".

2. **Security review.** Dispatch the `security-reviewer` agent over the whole changeset (not just the last section). Critical findings block completion. Major findings: fix or present to me with the tradeoff. One waiver exists, on a file-type predicate, not a judgment call: when every file in the changeset is prose (markdown or plain text - no code, script, hook, config, query, or schema file anywhere in it), the dispatch may be skipped, with the skip and the changed-file evidence recorded in the final Chapter. A single non-prose file, even a one-line edit, voids the waiver: run the review, scoped to the non-prose files when the rest is prose.

3. **Final adversarial review.** Dispatch the `adversarial-reviewer` agent over the entire changeset against the spec. Per-section reviews catch local issues; this pass catches cross-section cohesion problems, leftover debris (dead code, stale TODOs, orphaned files), and spec items that fell through the cracks.

4. **Documentation curation.** Dispatch the `docs-curator` agent with the spec path. It updates the project's docs/ from the as-built code and returns a Drift Report with each item tagged `Class: mistake | deviation`. A `mistake` (the code may be wrong, not just the docs) is a true blocker: stop, put it to me, and resolve it (a code, spec, or doc fix) before the PR. A `deviation` is the docs now reflecting a deliberate as-built choice: record it in the final Chapter and surface it in the PR description for awareness, with no stop. **Never silently reconcile a `mistake`; never halt the run for a `deviation`.** This fast-path means a clean or deviation-only effort flows straight through a long run, and only a likely-mistake stops it.

5. **Close and archive the plan doc.** Set `Status: Complete` and append a final Chapter summarizing the effort, the review outcomes, and the drift adjudications. Then invoke the `curating-docs` skill to finish the lifecycle: `git mv` the plan from `docs/plans/` into `docs/archive/` (history preserved, Chapters intact), act on any cross-reference gaps the docs-curator flagged, prune `docs/backlog.md` of items this effort completed into the quarter's archive snapshot, and refresh the `docs/README.md` index. A plan is not closed until it has left `docs/plans/`.

   **Finalizing the doc is not committing it.** This step runs in full under every commit model: flip to Complete, append the final Chapter, archive, and (Review-Only) stage it with the code. The Review-Only deferral is about the commit, which is mine, never about leaving the doc open. The resting state you hand me is a closed, staged plan, so my one review-commit lands the code and a Complete, archived doc together.

   | The excuse | Why it is wrong |
   |---|---|
   | "Review-Only means leave the doc open." | Review-Only defers the commit, not the truth. Flip the plan to Complete and stage it; I commit. |
   | "The plan is not done until it is accepted, so leave it In Progress." | If the work is delivered and the gates passed, the plan is Complete now. A change I request reopens it as a new round. |
   | "Flip it to Complete after the commit." | After the commit you are out of the loop and nothing flips it. That is exactly how a finished plan sits stale. Finalize before you hand off. |

6. **Apply the commit model.** For Branch-and-PR and Commit-and-Push, the docs work from steps 4 and 5 (the curator's edits, the archival `git mv`, the backlog prune, the index refresh) is committed *before* any push or PR, in the same commit series as the code. The docs ship in the same PR as the code, never as a follow-up: where neither author can release their own PR, a separate docs PR is a governance dead-end, so the docs must ride in the one authored PR. Do not open the PR, or do the Commit-and-Push final push, while the plan still sits in `docs/plans/` or `git status` shows uncommitted `docs/` changes. Red flags that you are about to strand the docs: "open the PR, clean docs after", "docs can be a follow-up PR", "the code is committed, tidy docs next". The predicate gates the PR (plan archived, `docs/` clean), not your read of whether the docs feel done. This applies to every durable record, not just docs: the Chapters, decision records, and the register are committed before the merge is requested, and the merge request is the effort's last action. A PR branch is frozen the moment it is up for merge: do not push to it again. Anything decided after that (a post-review decision, a late close-out note) goes in a separate doc PR opened last against the current integration branch, never back to the up or merged branch. Pushed is not merged.
   - **Review-Only:** present a consolidated walkthrough, every changed file, what changed and why, organized by section, with a diff summary (the staged changes are the review surface). Then stop; I review before anything is committed.
   - **Branch-and-PR:** with the docs commit already on the branch, verify the branch builds and tests green, update from origin and surface any sibling-session conflicts for resolution, push the branch, and open the pull request via host detection (GitHub `gh pr create`, Azure DevOps Git `az repos pr create`, or, when no CLI or auth is present, push and surface the "create a pull request" URL the host prints). The PR carries the title, a by-section summary, the test evidence, the archival summary, and the deviation drift items. Then present the options: merge, keep the branch for iteration, or discard. Never merge without my explicit choice; on my merge, offer the teardown below. After any merge (mine in-session, or detected on a later session), run the strand-check before trusting the records landed: `git fetch`, then `git log origin/<integration>..origin/<branch>`. Any commits there are post-merge work stranded off the trunk; recover them via a new doc PR against the integration branch, never by reopening the merged branch.
   - **Commit-and-Push:** the docs commit lands with the section commits. If the session ran directly in the main checkout, the commits already landed, so confirm everything, code and docs, is pushed. If concurrency put the session in a worktree on a feature branch: update from origin (surfacing conflicts rather than failing), merge to main, push, then **tear down: remove this session's worktree and delete its branch local and remote**. Teardown touches only the branch and worktree this session created, is reversible (the commits are already in main), and is named in the close-out.

7. **Bank the learnings.** Anything durable discovered during the effort (build quirks, conventions, gotchas, environmental facts) belongs in auto memory, not the plan doc. Save it now, while it is fresh.

8. **Kaizen check.** First make sure any kit friction from this effort (captured along the way, or a Chapter Surprise that traced to the kit) is in the kaizen inbox. Then offer a kaizen pass in one line only if the inbox has pending items; on a clean effort the inbox is empty and you say nothing. The predicate, not your read of the session, gates the offer.
